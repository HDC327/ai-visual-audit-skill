# Skill：静默反馈与自动优化协议

版本：1.0 | 作用于：`SKILL.md`、`skill-review-flow.md`、`skill-review-criteria.md`

> 本文件定义视觉建议后的轻量反馈闭环。用户只感知自然对话；Agent 在后台把认可、纠错、追问和执行违规沉淀为反馈信号，并在达到阈值后自动优化相关文档。优化后清空已消费反馈，避免项目无限膨胀。

---

## 1. 用户交互原则

- 每次完整视觉建议结尾都加一句轻量邀请，不暴露内部术语：
  `如果有哪里判断不准，直接告诉我，我会记下来把之后的判断调得更准。`
- 不要求用户按模板反馈，不展示“采纳 / 误报 / 漏报 / optimizer / criteria”等内部流程。
- 如果用户下一轮没有反馈，正常继续对话；不要追问“是否采纳”。
- 用户明确问反馈机制时，可以解释机制；否则静默执行。

---

## 2. 反馈推理分类

从用户后续对话中自动推理反馈类型：

| 反馈类型 | 触发信号 | 处理方向 |
|---|---|---|
| `positive` | “对”“准确”“有帮助”“就按这个改” | 记录有效判断，后续可作为稳定口径参考 |
| `false_positive` | “这个不对”“你看错了”“这里不是问题” | 降低过度敏感，必要时放宽 criteria 边界 |
| `missed_issue` | “你漏了”“还有更严重的”“应该先看这里” | 补足识别盲区，必要时收紧 criteria 或新增易错点 |
| `execution_violation` | “skill 允许吗”“不该打分”“格式不对”“你违反规则” | 优先修 `SKILL.md` 或 `skill-review-flow.md` 的执行自检 |
| `needs_explanation` | “为什么这么说”“依据是什么”“你怎么看出来的” | 不默认算负反馈；用于增强证据表达 |

同一轮用户消息可能包含多类反馈，保留最主要的一类；涉及合规、安全、版权时优先按 `missed_issue` 或 `execution_violation` 记录。

---

## 3. 静默写入

运行时反馈写入 `references/feedback/feedback-log.md`。每条记录包含：

- 日期与场景
- 用户原话摘要
- Agent 原判断摘要
- 推理出的反馈类型
- 影响的规则位置
- 建议优化方向
- 状态：`pending` / `consumed`

写入要求：

- 静默写入，不在普通对话中说明“已记录”。
- 不保存用户隐私、完整图片路径、账号、手机号等敏感信息；只保留足够复盘的文字摘要。
- 单条记录控制在 6 行以内。
- 如果无法写入文件，继续服务用户，不因反馈记录失败打断对话。

---

## 4. 自动优化阈值

达到以下条件时，Agent 可静默优化，不需要用户人工确认：

| 条件 | 自动动作 |
|---|---|
| 同类 `execution_violation` >= 3 条 | 更新 `SKILL.md` 或 `skill-review-flow.md` 的输出前自检 |
| 同类 `false_positive` >= 5 条 | 放宽对应 criteria 边界，或把问题从“必须先改”降为“建议优化/不展开” |
| 同类 `missed_issue` >= 5 条 | 在 `skill-review-criteria.md` 增加易错点或收紧边界 |
| `needs_explanation` >= 5 条且集中在同一维度 | 强化该维度的“为什么/依据”表达要求 |
| 总反馈 >= 30 条 | 做一次批量自动优化，生成优化摘要 |

高风险合规红线只允许自动收紧或增强提示，不允许因为少量用户反馈自动放宽。方向矛盾的反馈不自动改规则，只写入 `change-log.md` 的“待观察”。

---

## 5. 优化后清空策略

自动优化完成后：

1. 将本轮消耗的 `pending` 反馈汇总到 `references/feedback/change-log.md`。
2. 在 `feedback-log.md` 保留文件说明、当前空队列标题和最近一次优化摘要。
3. 删除或清空已消费的明细反馈，只保留未消费的 `pending` 条目。
4. 不保留超过 2 次优化周期的明细，避免仓库持续膨胀。

`change-log.md` 保留每次优化的摘要，而不是保存所有原始反馈。每次摘要最多 20 行。

---

## 6. 输出前自检

视觉建议发出前，Agent 必须快速检查：

- 没有数字评分、百分比评分或终审语言。
- 结尾有轻量反馈邀请。
- 用户没有要求专业审核口径时，不暴露内部反馈分类。
- 如果上一轮用户指出执行违规，本轮优先修正该违规，不继续沿用错误格式。
