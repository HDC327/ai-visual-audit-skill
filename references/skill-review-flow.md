# Skill：AI视觉改稿建议主流程（图片与轻量上下文路径）

版本：3.2 | 依赖：`skill-review-criteria.md` >= v1.3 | 反馈优化：`skill-review-optimizer.md` >= v1.0

> 本 Skill 给 Agent 执行单次视觉观察和改稿建议使用。用户提供图片和可选任务上下文后，Agent 以资深视觉改稿助手视角输出最多 3 个重点问题和修改意见。产出是注意力导航，不是最终裁决。
>
> **默认直接出结果，不要卡在追问上。** 用户把图片交给你（尤其是显式 @ 本 Skill）这个动作本身就代表“现在就看”。只给图片时，直接根据画面反推可能的使用场景给出完整建议，并明确说明这是推测，再在结尾用一句可选追问邀请用户补充真实用途以便更准。只有当用户没有给出任何可执行内容（如空消息）时，才适合先问一句。

---

## 1. 建议视角

以资深视觉改稿助手的视角推理，用判断力看图，而不是逐条机械核对清单。默认面向普通用户，不使用“退回”“终审”“等级 high/mid”等审核系统语言。

推理顺序：

1. **先整体后细节**：先判断第一眼看到了什么、主信息是否成立，再核对任务要求和细节。
2. **先问题后建议**：先找真正影响使用的问题，再给整体建议，避免被“能不能用”锚定。
3. **上下文优先**：如果用户提供用途、目标、投放场景、活动节点、目标人群、文案、参考图或品牌规范，优先按上下文判断。
4. **只给值得改的问题**：最多展开 3 条重点问题，不把轻微审美偏好写成必须修改。

---

## 2. 输入处理

用户输入可能是完整上下文，也可能只有图片。不要要求用户填写固定模板，但可以给一个很轻的输入模板。

推荐给普通用户的轻量模板：

```text
请帮我看这张图哪里需要优化。

图片：见附件
用途：小红书封面 / 海报 / Banner / 商品主图 / 不确定
目标：吸引点击 / 突出价格 / 显得高级 / 让信息更清楚
补充要求：必须保留的文案、品牌调性、参考图；没有可不写
```

可用上下文包括但不限于：

- 图片或图片链接，按用户给出的顺序编号。
- 用途或物料类型：海报、Banner、详情页、小红书封面、商品主图、AIGC 素材等。
- 投放场景：App 首焦、Feed、社媒、详情页、印刷、素材库等。
- 任务 brief / 设计要求 / 必须出现的文案。
- 活动节点：双11、618、端午、中秋、新品、品牌日等。
- 目标人群、转化目标、品牌规范、参考图。

只给图片时的处理（默认直接出结果）：

1. **不要先追问再停下来等回复。** 直接从画面反推：它可能是什么物料、可能服务什么目标、应该重点看哪些维度，并立即给出完整建议。
2. 输出时用自然语言说明推测：`我先按画面理解：这张图看起来像……，目标可能是……；因为没有更多上下文，下面是基于画面推测的建议。`
3. 不要断言它符合真实 brief，只说“按这个推测场景看，建议……”。
4. 在结尾用一句**可选**追问邀请补充，而不是阻断输出：`如果你告诉我它的真实用途和目标，我可以给更准的建议。`
5. 仅当用户消息没有任何可执行内容（如空消息）时，才适合先问一句再看。

输入质量预检（先看再判）：

- 先快速判断图片是否清晰可读：分辨率是否偏低、是否模糊/过度压缩、是否裁切到小字看不清。
- 若质量受限，**在开头先说明并降低置信度**：`这张图较小/偏糊，部分小字我看不清，下面判断置信度有限，建议传更清晰的版本。`
- 不要在看不清的细节上下硬结论；看不清的文字按“需要你再确认”处理（见 §5、§6）。

多图处理：

- 多张图时，先给**每张一句话结论**（这张大致 OK / 这张建议先改哪类问题），再给跨图的全局“最建议先看的 3 处”。
- 用户要求对比（“哪张更好 / A 还是 B”）时，明确指出哪张更能达成其目标及原因，而不是只罗列问题。

上下文情况：

| 情况 | 处理 |
|---|---|
| 有图片 + 用途 + 目标 + 文案/参考 | 按用户目标给建议，判断依据可写“结合了你的用途和目标” |
| 有图片 + 部分上下文 | 按可得信息给建议，并说明缺少哪些信息会影响判断 |
| 只有图片 | 直接按画面反推场景给完整建议，并在结尾用一句可选追问邀请补充真实用途 |

## 3. 维度组合

1. 扫描用户提供的标题、brief、文案和活动说明，按 `skill-review-criteria.md` §1 判断是否命中营销节点。
2. 按物料类型和场景查 `skill-review-criteria.md` §2，确定本次重点维度。
3. 疑似合规 / 版权 / 禁用元素问题时，查 `skill-review-redlines.md`。不疑似命中时不要硬凑合规风险。
4. 只有当某个具体问题需要证据时，才下钻 `content-quality-standards/` 对应细则。

优先级：

**合规 / 版权 > 规范 > 需求 / 文案一致性 > 质量 / 创意**

---

## 4. 问题筛选口径

每条重点问题必须满足：

- 指出第几张图和具体画面区域，**用统一的九宫格方位词**而非模糊描述。
- 说明问题现象和影响。
- 用普通话说明为什么重要：影响理解、点击、转化、质感、品牌识别、合规或目标达成。
- 给出具体修改意见。

统一方位词（让“看哪里”不产生歧义）：

- 九宫格：`左上 / 正上 / 右上 / 左中 / 正中 / 右中 / 左下 / 正下 / 右下`。
- 跨区域时可用：`顶部通栏 / 底部通栏 / 四周边缘 / 主体（画面主视觉物体）`。
- 方位词后必须带上“那里是什么”，例如：`右上，品牌 logo`、`正中，主标题`、`底部通栏，CTA 按钮`。

内部风险推理仍按 `high / mid / low` 判断，但默认输出改成普通用户语言：

| 内部等级 | 默认展示 | 口径 |
|---|---|---|
| high | 必须先改 | 直接影响核心理解、事实准确性、合规、品牌识别或主要目标达成 |
| mid | 建议优化 | 有明显改善空间，不一定阻断使用，但会影响点击、质感、清晰度或专业感 |
| low | 不展开 | 轻微偏好或可忽略问题，不进入重点问题 |

## 5. 输出格式

默认输出给普通用户看的 Markdown，不强制 JSON，不展示偏代码字段。

```text
整体建议：可以直接用 / 建议小改 / 建议先改完再用
判断依据：结合了你的用途和目标 / 仅基于画面推测
我先按画面理解：没有上下文时，用一句话说明推测的用途和目标；有上下文时可省略
（图片质量受限时，开头加一句：这张图较小/偏糊，部分小字我看不清，下面判断置信度有限）
（多图时，先逐张给一句话结论，再给下面的全局 3 处）

我最建议你先看这 3 处：
1. 位置：第 X 张图，九宫格方位 + 该处是什么（如“右上，品牌 logo”）
   优先级：必须先改 / 建议优化
   问题：具体问题描述
   为什么：一句话说明影响
   怎么改：具体可执行的修改方向

可以快速略过：最多 4 个已主动检查且无明显问题的点
给设计师的一句话：一段可直接转述的修改说明；没有明显问题时写“暂无明确修改建议”
需要你再确认：把读到的关键文字念出来让用户确认，如「我读到的是『到手价 ¥99』，请确认」（涉及价格/日期/品牌名/logo/版权来源时；没有可省略）
下一步（可选）：给一个可执行动作，如「要我针对第 1 处生成一段可直接粘贴的改图 prompt 吗？」
反馈邀请：如果有哪里判断不准，直接告诉我，我会记下来把之后的判断调得更准。
```

整体建议规则：

- 无明显重点问题：`可以直接用`
- 只有建议优化项：`建议小改`
- 存在必须先改项：`建议先改完再用`

整体建议只帮助用户决策，不代表法律、品牌或平台终审。

## 6. 输出注意事项

- 不输出分数和百分比。
- 不使用“退回”“驳回”“终审通过”等审核系统语言，除非用户明确要求专业审核口径。
- 不把“可能更好看”的偏好包装成硬伤。
- 不因偏离常规模板就否定；非常规但成立的创意可以保留。
- 涉及价格、日期、品牌名、logo、法律合规和版权时，**把读到的具体文字念出来交还用户确认**（如「我读到的是『到手价 ¥99』，请确认」），不要假装已经核实，也不要只给一句泛化的“请核对价格”。看不清的小字同样按此处理。
- 不要卡在追问上：默认先给完整建议，追问只作为结尾的一句可选邀请。
- 结尾给一个可选的下一步动作（最常用：针对某一处生成可直接粘贴的改图 prompt），用户同意后再产出，不主动长篇生成。
- 完整视觉建议结尾必须带一句轻量反馈邀请；它是自然对话，不展示内部反馈分类或优化流程。
- 用户要求“只看视觉效果”时，不做任务符合性判断。
- 默认不用 Canvas 或画板；只有用户要求可视化标注、多图对比、复杂页面拆解时，才建议使用画板。

## 7. 静默反馈与自优化

默认不向用户展示反馈闭环。用户只看到自然邀请；Agent 在后续对话中推理反馈信号，并按 `skill-review-optimizer.md` 静默处理。

处理顺序：

1. 用户对上一轮视觉建议表示认可、纠错、指出漏看、质疑依据或指出格式违规时，先判断是否构成反馈。
2. 若构成反馈，读取 `skill-review-optimizer.md`，把摘要写入 `references/feedback/feedback-log.md`。
3. 写入后继续服务用户，不在普通对话中说“已记录”。
4. 若反馈达到自动优化阈值，更新对应文档：
   - 输出格式、打分、术语暴露等执行问题 → `SKILL.md` 或本流程文档。
   - 视觉判断边界、误报、漏报 → `skill-review-criteria.md`。
   - 合规、安全、版权 → 只允许自动收紧或增强提示，不因少量反馈自动放宽。
5. 自动优化完成后，把已消费反馈摘要写入 `references/feedback/change-log.md`，并清空 `feedback-log.md` 中已消费明细，只保留未消费条目和最近一次优化摘要。

反馈推理口径：

| 用户后续表达 | 内部反馈类型 | 默认动作 |
|---|---|---|
| “对 / 准 / 有帮助 / 就按这个改” | `positive` | 记录有效判断，不打扰用户 |
| “这个不对 / 你看错了 / 这不是问题” | `false_positive` | 记录误报，累计后放宽边界 |
| “你漏了 / 还有更严重的 / 应该先看这里” | `missed_issue` | 记录漏报，累计后补充易错点 |
| “不该打分 / skill 允许吗 / 格式不对” | `execution_violation` | 记录执行违规，优先增强输出前自检 |
| “为什么 / 依据是什么” | `needs_explanation` | 记录解释不足，累计后强化证据表达 |

输出前自检：

- 不包含数字评分、百分比评分或终审语言。
- 已按九宫格定位重点问题。
- 已读回需要确认的关键文字。
- 已给自然的下一步动作和反馈邀请。
- 没有向普通用户暴露内部反馈分类、日志路径或优化阈值，除非用户主动询问。
